# 2026-06-29 使用舒适性与便捷性下一阶段完善建议

## 用户问题

现在的功能设计完成已经了基础的功能开发，接下来要对使用的舒适性，便捷性等方面进行开发，结合现在的功能帮我想一下下一步应该完善哪些方面。

## 当前代码观察

- 项目当前是 React 19 + Vite + Tauri 2 桌面应用，依赖包含 lucide-react、sonner、react-markdown、streamdown、radix-ui、tailwindcss，适合继续做桌面级交互体验、流式 AI 阅读体验和统一组件系统。
- 主链路已经形成：初始化设置、项目列表、工作区、设置页；工作区阶段包括大纲、人物、目录、正文。
- 侧栏已经承担项目入口、创作阶段入口、设置入口、当前模型状态、连接状态、过时状态等信息。
- AIContext 已经支持流式内容、thinking 内容、错误状态、生成阶段、取消生成等基础能力。
- 数据库目前包含模型预设、项目、大纲、人物、章节、正文、过时标记、写作风格配置、全局设置等核心表。
- 当前体验优化方向不应优先继续堆新业务模块，而应围绕“减少中断、减少认知负担、增强可控感、增强数据安全感”做一轮体验型迭代。

## 总体结论

下一阶段建议定义为：**V0.2 舒适性与便捷性增强版本**。

目标不是改变主功能，而是让已有的“大纲 → 人物 → 目录 → 正文”创作流程更顺、更稳、更像一个可以长期写作的桌面工具。

## P0：优先开发，直接影响日常使用舒适度

### 1. 全局布局与滚动体验统一

当前多个页面分别处理 `ScrollArea`、底部操作栏、左右分栏、固定宽度，后续容易出现不同页面体验不一致的问题。建议抽象统一的工作区布局组件。

建议新增：

- `WorkspacePageLayout`：统一标题区、状态区、内容区、底部操作栏。
- `EditorActionBar`：统一 AI 生成、停止生成、保存、模型选择、润色按钮样式。
- `ResponsiveSplitPane`：统一章节列表/正文编辑、目录列表/详情编辑这类左右分栏。
- `AppScrollArea`：在当前 `ScrollArea` 基础上封装尺寸、暗色滚动条、内边距、横向溢出处理。

验收标准：

- 大纲、人物、目录、正文四个页面滚动体验一致。
- 暗色模式滚动条不突兀。
- 小窗口下不会出现内容被遮挡、按钮挤压、右侧输入框不可见。
- 底部操作栏始终可见，主内容区域独立滚动。

### 2. AI 生成过程可控化

当前已经有流式生成、thinking、错误、取消生成等基础能力，但用户在长文本生成时仍需要更强的掌控感。

建议新增：

- 生成前确认：本次会覆盖还是追加？
- 生成中状态面板：当前阶段、当前模型、已生成字数、耗时。
- 停止后处理：保留已生成内容，可选择保存、丢弃、继续生成。
- 失败后处理：重试、复制错误、切换模型重试。
- 生成结果应用方式：替换当前内容、追加到末尾、另存为草稿。

验收标准：

- 用户不会因为误点 AI 生成而丢失已有内容。
- 网络失败、模型失败时有明确恢复路径。
- 停止生成后已有内容不会凭空丢失。

### 3. 自动保存与版本快照

当前大纲和正文已有生成完成后的自动保存逻辑，但编辑过程中的“未保存风险”仍比较高。建议把保存能力升级为全局体验能力。

建议新增：

- 编辑器输入 debounce 自动保存。
- 顶部统一显示：已保存 / 正在保存 / 未保存 / 保存失败。
- 最近 10 次快照，可恢复。
- AI 生成前自动生成一次快照。
- 润色前后生成对比快照。

数据库建议新增：

```sql
CREATE TABLE content_snapshots (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  project_id INTEGER NOT NULL,
  target_type TEXT NOT NULL,
  target_id INTEGER,
  content TEXT NOT NULL,
  reason TEXT NOT NULL,
  created_at TEXT DEFAULT (datetime('now'))
);
```

验收标准：

- 所有核心文本编辑都不依赖用户手动保存。
- AI 覆盖、润色、误删后可以恢复。
- 保存失败时用户能明显感知。

### 4. 下一步引导与创作流程提示

当前阶段导航已经存在，但更适合“切换功能”，还缺少“下一步该做什么”的产品引导。

建议新增：

- 工作区顶部显示当前阶段说明和推荐下一步。
- 大纲完成后提示：可以进入人物设计。
- 人物为空时禁止/弱化目录生成，并解释原因。
- 目录完成后提示：选择章节生成正文。
- 过时标记不仅提示“需要重新生成”，还说明“为什么过时”。

验收标准：

- 新用户不看文档也知道下一步该点哪里。
- 用户修改上游内容后知道哪些下游内容受影响。

## P1：第二阶段开发，提升效率和专业感

### 5. 项目管理增强

当前项目列表偏基础，适合早期验证，但长期使用会需要检索、归档和备份能力。

建议新增：

- 项目搜索。
- 最近打开排序。
- 重命名项目。
- 复制项目。
- 归档项目。
- 项目卡片显示：字数、章节数、更新时间、当前阶段。
- 项目导入/导出。

### 6. 人物功能区体验增强

人物是小说创作中高频编辑对象，建议把人物功能区做成“角色数据库”而不是简单列表。

建议新增：

- 人物按主角/配角/其他筛选。
- 人物卡片支持紧凑模式/详情模式。
- 展开后的字段全部使用自适应多行输入。
- 增加人物关系视图：A 与 B 的关系摘要。
- 支持拖拽排序。
- 支持批量生成缺失字段。
- 支持从正文/大纲中提取新人物。

### 7. 模型配置便捷性

当前模型配置可以添加 API 地址、Key、模型名，并支持获取模型列表，但可以进一步降低配置门槛。

建议新增：

- 常用服务商模板：OpenAI、OpenAI Compatible、本地代理、硅基流动、DeepSeek 等。
- API Key 只显示尾号，支持复制/重新输入。
- 测试连接按钮，明确区分“网络失败 / Key 错误 / 模型不可用”。
- 每个阶段可以设置默认模型，例如大纲用强推理模型，正文用长文本模型。
- 失败时一键切换备用模型重试。

### 8. 正文编辑体验增强

正文编辑是最终核心场景，应增加写作软件常见能力。

建议新增：

- 专注模式。
- 全屏写作。
- 字数目标。
- 当前章节字数、总字数统计。
- AI 味高频词点击替换。
- 润色前后 diff 对比。
- Markdown/纯文本预览切换。
- 自动滚动到生成位置。

## P2：第三阶段开发，提升完整产品能力

### 9. 导出与发布能力

建议新增：

- 导出 TXT。
- 导出 Markdown。
- 导出 DOCX。
- 按章节合并导出。
- 导出项目包，用于迁移和备份。

### 10. 本地数据安全与维护能力

建议新增：

- 数据库备份。
- 数据库恢复。
- 启动时检测数据库损坏。
- 本地日志查看。
- 一键打开数据目录。
- 清理缓存/临时生成内容。

### 11. 新手模板与创作模板

建议新增：

- 小说类型模板：玄幻、科幻、悬疑、都市、言情、历史。
- 大纲模板。
- 人物模板。
- 章节节奏模板。
- 写作风格模板。

## 推荐开发顺序

### Sprint 1：体验基础设施

- 抽象 `WorkspacePageLayout`。
- 抽象 `EditorActionBar`。
- 封装 `AppScrollArea`。
- 修复所有页面的小窗口自适应问题。
- 统一暗色滚动条和分栏滚动体验。

### Sprint 2：AI 生成可控化

- 增加生成前覆盖确认。
- 增加停止后的保存/丢弃/继续选择。
- 增加错误重试与切换模型重试。
- 增加生成记录表。

### Sprint 3：保存与版本安全

- 增加统一自动保存。
- 增加快照表。
- AI 生成和润色前自动快照。
- 增加恢复入口。

### Sprint 4：编辑效率增强

- 项目搜索、重命名、归档。
- 人物筛选、关系视图、批量补全。
- 正文专注模式、字数目标、diff 对比。

### Sprint 5：导出与数据维护

- TXT/Markdown/DOCX 导出。
- 项目包导入导出。
- 数据备份恢复。
- 日志与问题诊断页。

## 架构建议

前端建议形成以下分层：

```text
src/components/layout       全局布局与窗口结构
src/components/editor       编辑器通用组件
src/components/ai           AI 生成状态、结果应用、错误恢复
src/components/project      项目管理组件
src/components/characters   人物数据库组件
src/components/shared       通用提示、状态、空态、滚动容器
```

后端建议新增以下能力：

```text
commands/snapshots.rs       快照保存、查询、恢复
commands/export.rs          导出 TXT/Markdown/DOCX
commands/backups.rs         数据库备份、恢复、打开目录
commands/generation_logs.rs AI 生成记录与失败诊断
```

建议新增数据库表：

- `content_snapshots`：保存快照。
- `generation_logs`：保存 AI 生成记录、模型、耗时、错误。
- `project_preferences`：保存项目级偏好，如默认模型、显示模式。
- `recent_projects`：记录最近打开项目。

## 最终判断

现在最值得做的不是继续扩大功能边界，而是把已有功能打磨到“能长期舒服地写”的程度。

优先级应该是：

1. 布局、滚动、自适应统一。
2. AI 生成过程可控。
3. 自动保存和版本快照。
4. 下一步引导和过时原因解释。
5. 人物、目录、正文的编辑效率增强。
6. 导出、备份、恢复。

这样做可以让 OpenCodeWriter 从“基础功能可用”进入“真正适合长时间创作”的阶段。
