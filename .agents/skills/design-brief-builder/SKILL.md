# 构建设计规范

## 输入

完整读取 `Product-Spec.md` 和已有设计资料。需要澄清抽象视觉词时，读取 [references/design-decisions.md](references/design-decisions.md)。

## 原则

- 先确定产品形态，再讨论视觉；网页、桌面应用、终端工具和对话式 Agent 使用不同交互模型。
- 不问“想要什么风格”；给出少量具体方向及其取舍。
- 把抽象词翻译成间距、密度、颜色、字重、层级、动效等属性，并让用户确认。
- 让设计服务任务优先级，不用装饰掩盖信息结构问题。

## 产出

创建或更新 `Design-Brief.md`，包含产品形态、设计目标、信息架构、页面/视图清单、状态矩阵、视觉令牌、组件原则、响应式或窗口行为、无障碍要求和明确禁用项。

## 验收

- Product Spec 中每个有界面的功能都映射到页面或状态。
- 每个页面覆盖默认、空、加载、错误和关键成功状态。
- 所有视觉判断可转成具体设计属性，不依赖“高级感”等未定义词。
- 冲突时记录权威顺序：设计稿高于 Design Brief，Design Brief 高于 Product Spec 的视觉描述。

## 停止条件

产品形态、核心任务或目标平台未确定时停止，请用户先拍板，不生成虚假的完整规范。
