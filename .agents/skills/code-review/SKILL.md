# 审查代码

## 输入

获取完整 Spec 条目、设计资料、DEV-PLAN 交付清单、变更 diff、涉及文件和验证输出。使用独立上下文，不继承实现者的推理摘要。

## 原则

- 只根据需求、代码和运行证据判断，不替实现者补全意图。
- 每个结论要么有证据，要么不成立；严重级别由实际影响决定。
- 只报告不修复，保持审查上下文与实现上下文隔离。
- 区分本次引入问题和预先存在问题，不扩大当前修复范围。

## Stage 1：做对了没有

- 逐条映射需求到实现文件和行号。
- 检查遗漏功能、错误行为、状态覆盖和范围外实现。
- 检查界面引导真实性：提示、按钮和入口必须对应真实能力。
- 有高优先级正确性问题时停止，不进入 Stage 2。

## Stage 2：做好了没有

- 检查正确性、安全、数据边界、错误恢复、性能和可维护性。
- 检查测试真实性：断言必须证明行为，不得用假前提覆盖缺陷。
- 有设计稿时逐项核对布局、状态、视觉令牌和交互。
- 区分本次引入问题与预先存在问题；后者只单独报告。

## 产出

按严重级别列出发现，每项包含证据、文件行号、影响和失败标准。最后分别给出 Stage 1 与 Stage 2 的 PASS/FAIL。

## 验收

- 每个结论都有代码、测试、文档或运行结果证据。
- “大致符合”“应该没问题”不算结论。
- 无发现时明确说明已检查范围和残余风险。
- 审查员不修改代码、不提交、不替实现者修复。
- Stage 1 与 Stage 2 均 PASS 后，由主 Agent 在完成修复后验证的前提下清除 `.codex/.state/review-needed`；审查员不直接改状态。

## 停止条件

缺少 Spec、diff 或无法读取关键文件时停止并请求完整上下文。
