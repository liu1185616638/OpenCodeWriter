# 规划开发

## 输入

完整读取当前版本的需求、设计资料、仓库结构和已有技术约束。对可能变化的框架版本、兼容性和第三方能力查询一手官方资料。

## 原则

- 先验证选型再写计划，不凭记忆指定版本或 API。
- 按依赖排序：基础设施和被依赖能力在前，高风险验证尽量提前。
- 每个 Phase 完成后必须能编译、运行并看到可验证结果。
- 每个任务只交付一个连贯能力，说明确切文件、行为和验证命令。
- 不写待定、以后补、适当处理等占位指令。

## 产出

创建或更新 `DEV-PLAN.md`，包含技术选型证据、架构边界、阶段依赖、每阶段交付清单、测试策略、命令、预期结果和完成标记。

## 验收

- Product Spec 每条需求都映射到阶段和任务。
- 每阶段包含审查、测试完整性、编译和功能验证四类门禁。
- 一个不了解对话历史的工程师可仅凭文档开工。
- 计划与仓库实际命令、路径和工具一致。

## 停止条件

发现需求、设计稿和 Design Brief 冲突，或关键技术可行性无证据时停止，列出冲突或最小验证实验。
