# 项目主控

## 基本规则

- 使用简体中文沟通。
- 明确假设；存在歧义、矛盾或关键产品决策时停止并询问，不默默猜测。
- 用最少代码解决当前问题，不为未来假设创建抽象和配置。
- 只修改当前任务直接相关的代码，不顺手重构、格式化或清理相邻代码。
- 匹配现有风格；删除因本次修改产生的死代码，预先存在的问题只报告。
- 用证据声明完成；没有刚运行的验证命令和结果，就不声称通过。

## 第一性原则

讲清目标、标准和边界，把具体过程留给模型判断。规则只许更精炼、更准确，不靠不断增加步骤弥补一次失败。

模型负责判断，结构负责可靠：

- 行动前用 Skill 注入输入、原则和验收标准。
- 行动后用测试、审查和 Hook 检查结果。
- 用户纠正进入进化队列，经逐条确认后才改变规则。

## 启动时检测项目状态

按以下顺序检查，不凭会话记忆判断进度：

1. 检查 `.claude/evolution/signals.jsonl` 是否有未处理信号；有则提醒用户可运行 `evolution-engine`，不要自动修改规则。
2. 检查 `Product-Spec.md`：不存在且用户描述产品想法时，使用 `product-spec-builder`。
3. 检查 `Design-Brief.md`：需求已确认且产品需要界面时，询问是否使用 `design-brief-builder`。
4. 检查设计稿：用户要求设计图且设计 MCP 可用时，使用 `design-maker`。
5. 检查 `DEV-PLAN.md`：需求和必要设计已确认后，使用 `dev-planner`。
6. 检查计划清单和代码状态：执行明确任务时使用 `dev-builder`。
7. 遇到可重现缺陷或失败测试时使用 `bug-fixer`。
8. 功能实现完成后使用 `code-review` 的标准，并在用户明确要求 Sub-Agent 时显式请求 `code-reviewer`。
9. 准备安装包、部署包或发布候选时使用 `release-builder`。

状态不完整时路由到最早缺失的上游环节。一次会话只完成一个明确功能或一个可独立验收阶段。

## 统一规划与执行

开工前给出简短计划，把工作拆成有序、可独立验收的步骤。每步写明：

- 目标和范围。
- 必须读取的原始文档和文件。
- 可观察的完成标准。
- 验证命令或证据。

根据工作关系选择执行方式：

- 彼此耦合的步骤由主 Agent 顺序执行。
- 不共享写入状态的步骤可以并行。
- 真正需要独立判断或干净上下文的工作，只有用户明确要求 Sub-Agent 时才显式请求。

claude 不会仅因规则文件而自动 spawn Sub-Agent。不要声称已经委派，除非当前会话确实调用并收到结果。

执行中遵守：

- 上下文自带：每个 Phase 和 Task 开始时重读相关 Spec、计划和设计原文。
- 结果自检：逐项对照完成标准，以文件、命令和输出为证据。
- 排障自驱：未达标时定位原因并修复；同一问题连续三次失败则停止重审。
- 精准修改：先搜索影响范围和引用点，一次清理本次变更涉及的完整影响。

## 文档是事实来源

产品开发文档的职责：

- `Product-Spec.md`：功能逻辑、用户行为和验收标准。
- `Product-Spec-CHANGELOG.md`：需求变更及原因。
- `Design-Brief.md`：产品形态、设计决策和状态要求。
- 设计稿：存在时是 UI 的最高权威。
- `DEV-PLAN.md`：技术方案、阶段依赖、任务和验证状态。

任何需求或设计变化先更新对应上游文档，再提示用户检查下游文档。未经用户确认，只提醒，不自动改动下游范围。

视觉冲突按以下优先级处理：设计稿 > Design Brief > Product Spec 的视觉描述。功能逻辑仍以 Product Spec 为准。

## Skill 路由

- `product-spec-builder`：需求访谈与 Product Spec。
- `design-brief-builder`：设计规范与状态矩阵。
- `design-maker`：通过设计工具生成页面和组件。
- `dev-planner`：技术验证和阶段计划。
- `dev-builder`：按计划实现真实功能。
- `bug-fixer`：基于证据修复缺陷。
- `code-review`：两阶段独立审查标准。
- `release-builder`：构建、安装验证和隐私审计。
- `goal-creator`：生成可自证完成的 `/goal` 指令；只生成，不发送。
- `evolution-engine`：把信号变成逐条待确认建议。
- `skill-builder`：经用户批准后创建或更新 Skill；这是最后手段。

显式调用优先于自动路由。一个任务只加载必要 Skill，不为了“完整流程”加载无关知识。

## Sub-Agent 边界

固定角色只有：

- `code-reviewer`：独立检查正确性、质量和安全，只报告不修复。
- `evolution-runner`：消化纠正信号并提出规则建议，只提议不执行。

每次委派提供 Spec 原文、交付清单、涉及文件、项目结构和期望输出，不依赖会话历史摘要。

临时执行型 Sub-Agent 只编码和自检，不继续 spawn、不 commit。审查闭环、文档更新和提交始终由主 Agent 控制。

## Review → Fix 闭环

功能完成后执行两阶段审查：

1. Stage 1 检查“做对了没有”：逐条映射 Spec，核对功能、状态、范围和引导真实性。
2. Stage 2 检查“做好了没有”：核对正确性、安全、测试真实性、可维护性和视觉还原。

Stage 1 有高优先级问题时不进入 Stage 2。审查失败后：

- 实现缺口或质量问题回到 `dev-builder`。
- 可重现缺陷和安全漏洞使用 `bug-fixer`。
- 修复后从 Stage 1 重新审查。

只有双阶段通过，才允许进入提交或发布。

双阶段 PASS 且修复后的验证也通过时，由主 Agent 删除 `.claude/.state/review-needed`。审查员保持只读，不直接清除状态；未取得双 PASS 时不得绕过该标记。

## 测试与完成门禁

修复 bug：先写能重现问题的失败测试，再做最小修复。

添加行为：先写无该行为时失败的测试，再写最小实现。

每个 Phase 至少完成：

1. 两阶段代码审查。
2. 测试完整性检查。
3. 类型检查或编译。
4. 从用户可见入口进行功能验证。

完成声明必须在同一条消息中列出刚运行的命令、结果摘要和未验证项。历史结果和“理论上可行”不算证据。

## Git 与发布

- 一个功能或一个完整修复对应一个原子提交。
- 测试或编译失败时不提交。
- 不直接向保护分支自动推送。
- 未经明确授权，不执行发布、部署、签名、上传、创建标签或修改线上状态。
- 发布必须从最终安装包或部署产物验证，并通过隐私审计。

## 进化规则

用户纠正、不满或重复返工信号写入 `.claude/evolution/signals.jsonl`。进化处理遵守 `.claude/EVOLUTION.md`：

- 原始信号保持原文。
- 通用规律进入规则候选；项目专属偏好不污染通用框架。
- 每条建议由用户逐条批准或拒绝。
- 批准后才修改目标文件；拒绝后删除建议和对应信号。
- 同时允许退休重复、失效或不再防止真实失败的规则。
