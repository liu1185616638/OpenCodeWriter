# Product Spec — OpenCodeWriter

## 问题

小说创作者使用 AI 辅助写作时面临三重困境：

1. **AI 味太浓**：生成的文字语言套路化（高频词堆砌）、人物行为不一致（OOC）、情节推进机械化，缺乏拟人化与情感张力。
2. **流程断裂**：现有 AI 写作工具是泛类型的对话式工具，无法将大纲→人物→章节目录→正文这条递进式创作链路串起来，上游修改无法自动通知下游。
3. **专业性缺失**：没有专门的小说创作 AI 程序——没有创作方法论注入、没有模板约束、没有文档驱动的上下文管理，AI 只能凭感觉写作。

## 目标用户

首版服务以下四类人群：

| 用户类型 | 场景 | 核心诉求 |
|---------|------|---------|
| 写作小白 | 有故事想法但缺乏创作经验 | 引导式流程，开箱即用，降低创作门槛 |
| 网文爱好者 | 日更压力下需要高效产出 | AI 加速生成，保持风格一致性 |
| 专业小说作者 | 长篇创作中需保持人物和情节连贯 | 人物约束注入，上下文递进积累 |
| 编剧 | 剧本大纲和人物关系管理 | 结构化模板，层级清晰 |

## 用户场景

### 场景 1：小白首次创作
1. 打开程序，引导式配置向导帮助填写 AI API 地址和 Key
2. 创建新项目，系统提示必须先编写大纲
3. 在大纲编辑区输入故事设想，点击"AI 辅助生成"，AI 基于方法论+模板+示例生成大纲草稿
4. 用户修改大纲直至满意，进入人物阶段
5. 系统自动加载大纲内容，AI 基于大纲+人物模板生成人物小传（主要角色→重要配角→其他角色）
6. 继续推进章节目录→正文，每步自动携带上游上下文
7. 正文生成后使用润色 Skill 进行去 AI 味处理

### 场景 2：专业作者打磨作品
1. 在已有项目的正文阶段打磨某一章
2. 发现某人物行为不合理，切换到人物面板调整人设
3. 系统自动标记受影响的章节为"过时"
4. 作者选择对过时章节重新生成，或标记为"最新"保留当前版本
5. 继续在正文阶段逐章生成和润色

### 场景 3：多项目切换
1. 作者同时维护多个项目
2. 通过左侧项目导航切换项目
3. 每个项目的创作进度独立保存，切换后自动恢复到上次编辑位置

| F3 | 人物小传生成：基于大纲，严格模板层级 | P0 |
| F4 | 章节目录生成：基于大纲+人物 | P0 |

| F6 | Skills 系统：加载和执行预定义 Skill | P1 |

### 首版明确不做

- Skill 的可视化管理界面（创建/编辑/删除 Skill 的 UI）
- 内置免费模型端点（用户需自备 API Key）
- 多人协作
- 云同步
- 导出为 ePub/PDF 等格式
- 版本控制（Git 集成等）
- 自动重新生成下游内容（仅标记，不自动重生成）
- 移动端支持

## 功能要求

### F1 项目管理

**输入**：用户在项目列表界面操作
**行为**：
- 创建项目：输入项目名称，系统创建项目数据记录和目录结构
- 切换项目：左侧导航选择项目，右侧显示该项目当前阶段的编辑界面
- 删除项目：二次确认后删除项目及所有关联数据
**结果**：项目列表更新，切换后显示对应项目内容
**异常**：项目名称为空或重名时提示错误

### F2 大纲编写

**输入**：用户在编辑区输入故事设想，或点击"AI 辅助生成"
**行为**：
- 手动编辑：直接在编辑区编写大纲
- AI 生成：系统注入（创作方法论 + 大纲模板 + 大纲示例）作为上下文，调用 AI 生成大纲草稿
- 生成后用户可继续手动修改
- 大纲保存后，系统检查是否有已生成的人物/目录/正文，如有则标记为"过时"
**结果**：大纲内容持久化到数据库
**异常**：AI 调用失败时显示错误信息，不覆盖已有内容

### F3 人物小传生成

**输入**：用户点击"生成人物"或手动编辑人物面板
**行为**：
- 系统自动加载当前项目大纲作为上下文
- AI 基于大纲 + 人物模板生成人物小传，严格遵循层级：主要角色→重要配角→其他角色
- 每个角色包含：姓名、身份、外貌、性格、动机、人物关系、关键事件等（由模板定义）
- 用户可手动修改、新增、删除角色
- 人物保存后，系统检查是否有已生成的目录/正文，如有则标记为"过时"
**结果**：人物数据持久化，层级结构完整
**异常**：大纲不存在时阻止操作，提示"请先完成大纲"

### F4 章节目录生成

**输入**：用户点击"生成章节目录"
**行为**：
- 系统自动加载当前项目大纲 + 人物小传作为上下文
- AI 基于上下文 + 章节目录模板生成章节列表
- 每章包含：章节号、章节标题、章节摘要
- 用户可手动调整章节顺序、增删章节
- 目录保存后，系统检查是否有已生成的正文，如有则标记为"过时"
**结果**：章节目录持久化
**异常**：大纲或人物不存在时阻止操作，提示"请先完成上游阶段"

### F5 正文生成

**输入**：用户选择某一章节，选择模型预设，点击"生成正文"
**行为**：
- 系统自动加载大纲 + 该章出场人物小传 + 章节目录 + 写作风格配置作为上下文
- AI 基于上下文 + 正文模板生成该章正文
- 生成前自动执行去 AI 味预处理：注入风格参考文本、加载人设约束卡片
- 生成后可使用润色 Skill 进行二次加工（多轮迭代润色）
- 用户可手动编辑生成的正文
- 不满意时可切换模型预设重新生成，新结果替换旧结果
**结果**：该章节正文持久化
**异常**：上游阶段未完成时阻止操作；AI 调用失败时保留上次内容

### F6 Skills 系统

**输入**：Skill 定义文件（预定义格式，存放于 skills 目录）
**行为**：
- 程序启动时扫描 skills 目录，加载所有合法 Skill 定义
- 在对应创作阶段显示可用 Skill 列表
- 用户选择 Skill 后，系统按 Skill 定义注入上下文并调用 AI
- Skill 执行结果写入对应文档或展示在编辑区
**结果**：Skill 执行完成，结果可观察
**异常**：Skill 定义格式错误时跳过并记录日志；AI 调用失败时提示错误

### F7 写作风格配置

**输入**：用户在设置界面配置
**行为**：
- 可上传风格参考文本片段（喜欢的作品段落），长度上限 2000 字（超出截断并提示）
- 可编辑 AI 味高频词过滤/替换表（系统内置默认词库，用户可增删条目，词库随程序版本更新，用户也可手动添加新词）
- 可配置拟人化参数（如叙述视角、语言正式度、情感浓度等）
- 配置以项目为单位保存，不同项目可有不同风格
**结果**：风格配置在正文生成和润色时自动注入
**异常**：无配置时使用系统默认值

### F8 文档驱动上下文

**输入**：用户进入任意创作阶段
**行为**：
- 每个阶段开始前，系统自动从数据库读取已有文档：
  - 大纲阶段：方法论 + 模板 + 示例
  - 人物阶段：大纲 + 人物模板
  - 章节阶段：大纲 + 人物 + 目录模板
  - 正文阶段：大纲 + 人物 + 目录 + 写作风格 + 正文模板
- 读取的内容作为 AI 调用的上下文自动注入，用户无需手动指定
**结果**：AI 始终基于完整上游上下文生成内容
**异常**：上游文档为空时使用阶段默认模板

### F9 过时标记

**输入**：用户修改了大纲或人物
**行为**：
- 修改保存时，系统自动计算受影响的下游阶段
- 在左侧导航和编辑区以视觉标记显示"过时"状态
- 用户可选择"重新生成"（基于最新上游重新调用 AI）或"标记为最新"（保留当前内容，清除过时标记）
**结果**：过时状态准确反映数据一致性
**异常**：无

### F10 AI 模型配置

**输入**：首次启动或用户主动进入设置
**行为**：
- 引导式配置向导：依次填写 API 地址、API Key、模型名称
- 支持配置多个模型预设（如"创意写作用 Claude"、"快速生成用 GPT-4o"）
- 每次生成操作（大纲/人物/目录/正文/润色）前可选择使用哪个模型预设
- 不满意生成结果时，可切换模型预设重新生成同一内容，新结果替换旧结果
- 配置持久化到数据库
**结果**：AI 调用时使用用户配置的模型
**异常**：API 不可达时提示连接失败，建议检查配置

### F11 内置创作资源

**输入**：程序内置
**行为**：
- 随程序分发一套默认创作资源：
  - 创作方法论文件（指导大纲构建、人物塑造、情节设计的专业方法）
  - 各阶段模板文件（大纲模板、人物模板、章节模板、正文模板）
  - 各阶段示例文件（优秀大纲示例、人物小传示例等）
  - 默认写作风格文件
  - AI 味高频词库
- 资源文件存放在程序安装目录的 resources/ 下
**结果**：每个创作阶段自动加载对应资源
**异常**：资源文件缺失时使用最小内置 fallback

### F12 去 AI 味机制

**输入**：正文生成/润色阶段自动触发
**行为**：
- **风格参考注入**：将用户配置的风格参考文本加入 AI 上下文，引导模仿目标风格
- **高频词过滤**：生成后扫描 AI 味高频词，标记并建议替换
- **人设约束注入**：每章生成前自动加载该章出场人物的人设卡片，在 prompt 中强制约束"人物行为必须符合以下人设"
- **润色 Skill**：作为可卸载 Skill 提供，不内置到核心程序；安装后可执行多轮润色流程"去套路化→增强情感张力→检查人设一致性"
**结果**：生成文本的 AI 味显著降低，人物一致性提高
**异常**：无风格参考时跳过风格注入，使用默认

## 创作流程约束

```
新项目 → [大纲] → [人物] → [章节目录] → [正文]
          ↓         ↓          ↓           ↓
        必须先完成  必须先完成  必须先完成   逐章生成
                               ↑           ↑
                          章节打磨时    正文打磨时
                          可调大纲人物  可调大纲人物
```

- **首次创建**严格线性：大纲→人物→目录→正文，前一步未完成不能进入下一步
- **打磨阶段**自由编辑：在章节或正文阶段，可随时返回调整大纲和人物
- 上游修改后，下游自动标记为"过时"
- 每个步骤都可以是 AI 生成 + 用户手动调整的组合

## 技术约束

| 约束 | 选择 | 理由 |
|------|------|------|
| UI 框架 | OpenTUI + TypeScript + bun | 用户指定，原生性能 + 组件化架构 |
| 数据库 | SQLite | 单文件便携，适合桌面应用 |
| AI 接口 | OpenAI 兼容 API | 覆盖 OpenAI/Claude/国产模型等主流厂商 |
| 布局 | 分屏面板（左侧导航 + 右侧编辑） | IDE 式体验，信息密度高 |

## 失败路径

| 场景 | 处理 |
|------|------|
| AI API 调用失败 | 显示错误信息，不覆盖已有内容，支持重试 |
| AI 生成内容为空 | 提示生成失败，建议调整 prompt 或检查 API |
| 项目数据损坏 | SQLite 事务保护，启动时校验数据完整性 |
| 资源文件缺失 | 使用最小内置 fallback，日志记录缺失项 |
| Skill 定义格式错误 | 跳过该 Skill，日志记录错误详情 |
| 用户在未完成上游时试图进入下游 | 阻止操作，明确提示需要先完成哪个阶段 |

| A2 | 大纲生成后，AI 调用时上下文包含方法论+模板+示例 | 查看 AI 请求日志 |
| A3 | 人物小传包含"主要角色→重要配角→其他角色"三层级 | 检查生成结果结构 |
| A4 | 章节目录生成后，每章有标题和摘要 | 界面观察 |
| A5 | 正文生成时，AI 上下文包含大纲+人物+目录+风格配置 | 查看 AI 请求日志 |
| A6 | 修改大纲后，已有人物/目录/正文显示"过时"标记 | 界面观察 |

| A10 | Skill 可被加载、执行，结果可观察 | 执行 Skill 并检查输出 |

| A15 | 风格参考文本超过 2000 字时截断并提示 | 输入超长文本验证截断行为 |

## 已决决策

1. **润色 Skill 为可卸载 Skill**：不内置到核心程序，作为 Skill 文件分发，用户可自定义润色流程，也可卸载
2. **模型选择为每次生成可选**：不绑定全局或阶段，每次生成操作（大纲/人物/目录/正文/润色）前都可选择模型预设，不满意可换模型重新生成
3. **风格参考文本长度上限 2000 字**：超出截断并提示，避免挤占上下文窗口
4. **高频词库更新机制**：系统内置默认词库随程序版本更新，用户可手动增删条目扩展词库

- 不做云同步
- 不做导出为 ePub/PDF
- 不做版本控制集成
- 不做移动端
- 不做内置免费模型端点
- 不做 Skill 可视化管理界面
- 不做自动重新生成下游内容
